# Предложенные задачи по итогам быстрого ревью

## 1) Задача на исправление опечатки
**Проблема:** в README используется гибридное написание `backend-платформы`; для единообразия технической терминологии в проекте лучше использовать форму `back-end` (или полностью русскую форму `бэкенд-платформа`).

**Где найдено:** `README.md` (описание проекта).

**Задача:** привести написание термина к единому стилю по всему репозиторию (`back-end`/`бэкенд`), обновив README и связанные документы.

**Критерии готовности:**
- термин в README и документации использован в одном стиле;
- нет смешанного написания `backend`/`back-end` в соседних разделах.

---

## 2) Задача на исправление ошибки
**Проблема:** в конфигурации словаря `REL_TASK` upsert выполняется через `merge ... key(id)`, то есть ключ не включает `tenant_id`. Это может приводить к межтенантной коллизии при одинаковом `id` у разных арендаторов.

**Где найдено:** `refdata-shared/src/main/resources/application.yml` (блок `refdata.dictionaries[REL_TASK].apply.upsert-sql`).

**Задача:** изменить стратегию upsert для `REL_TASK` так, чтобы уникальность учитывала и `tenant_id` (например, `key(id, tenant_id)` для H2-режима и/или эквивалент в SQL для PostgreSQL).

**Критерии готовности:**
- upsert не затирает данные другого tenant при совпадающем `id`;
- добавлен регрессионный тест на два tenant с одинаковым ключом.

## 3) Задача на исправление комментария/документации
**Проблема:** OpenAPI-контракт описан неполно для query endpoint'ов: для `/items`, `/all`, `/version` отсутствуют параметры, заголовки и детальные ответы, хотя в контроллере и реализации они поддерживаются (`keys`, `X-Min-Version`, `X-Dict-Version`, `X-Data-Source`, `409`).

**Где найдено:**
- `docs/openapi.yaml`;
- `refdata-shared/src/main/java/com/contdistrapp/refdata/api/QueryController.java`.

**Задача:** синхронизировать `docs/openapi.yaml` с фактическим API query-сервиса.

**Критерии готовности:**
- у всех query endpoint'ов описаны path/query/header параметры;
- задокументированы response headers и `409 VERSION_NOT_COMMITTED`;
- примеры ответов соответствуют текущим DTO.

---

## 4) Задача на улучшение теста
**Проблема:** интеграционный Kafka-тест проверяет факт публикации в топик по `eventId`, но не валидирует ключ сообщения (partition key) и tenant/dict семантику routing, что не покрывает важный контракт упорядочивания.

**Где найдено:** `refdata-shared/src/test/java/com/contdistrapp/refdata/RelationalKafkaTopicTests.java`.

**Задача:** усилить тест: дополнительно проверять `record.key()` и структуру payload (tenantId, dictCode, eventType), а также добавить негативную проверку на отсутствие ложных совпадений.

**Критерии готовности:**
- тест падает при нарушении key-based routing;
- тест явно проверяет ключ и обязательные поля команды;
- исключены случайные passing-сценарии по частичному `contains(eventId)`.
